# 自动点技术点虚拟列表重构文档

## 范围

本轮只重构 `FH6SkillPointOcr.exe` 的自动点技术点找车流程。

暂不重构：

- `FH6VehicleDeleteOcr.exe` 自动删车流程。
- `FH6FullAuto.exe` 中找状态 `5` 的自用车辆流程。
- OCR 识别和格子映射规则本身。
- 固定点技能点按键序列。

原始意图继续以 `docs/flow-spec.md`、`docs/virtual-grid.md`、`docs/implementation-plan.md` 为准。本文件只描述这次重构后的控制逻辑。

## 当前问题

现有自动点技术点流程有两个找车入口：

1. `MainLoopOnce()` 进入斯巴鲁列表后先调用 `FindFirstVehicle()`。
2. 然后再调用 `FindValidNewCell()`。

这导致两个函数都会观察屏幕、写虚拟表、判断边界、滚动列表。结果是决策权分散，出现“首屏已经有状态 `3`，但先滚动后继续 OCR”“没有按最左最上的目标车处理”等问题时，只能继续小补丁，逻辑会越来越乱。

## 稳定对象

- `GridGeometry`：只负责当前电脑上的可见车辆格子像素区域。
- `VirtualVehicleList`：只负责当前运行中的全局车辆表、当前滚动 offset、状态查询和写表。
- `BuildVehicleGridObservation()`：只负责把一次 OCR 结果映射成车辆状态集合。
- `ApplyVehicleGridObservation()`：只负责把一次观察写入虚拟表。
- `SkillPointPlanner`：自动点技术点阶段唯一的下一步决策者。
- `ActionExecutor`：沿用现有 Runtime 输入方法，执行滚动、OCR、选中、停止。

## 动作集合

自动点技术点找车循环每轮只能选一个动作：

```text
Select(localCell)
Scroll(ticks, reason)
Observe(reason)
StopWithReset(reason, message)
```

含义：

- `Select`：当前可见窗口内已经有状态 `3`，直接用方向键移动并上车。
- `Scroll`：当前不应该 OCR 或选车，只需要滚动到下一个更有价值的位置。
- `Observe`：当前虚拟表不足以判断，需要 OCR 当前车辆列表区域并写表。
- `StopWithReset`：确认没有可点目标，先 `Esc` 复位到标准位，再停止。

## 决策顺序

每一轮先看虚拟表，再决定动作：

```text
if 当前可见范围有状态 3:
    Select(最左最上的状态 3)

else if 当前可见范围没有 3，但虚拟表后面已知有状态 3:
    Scroll(滚到能看到该状态 3)

else if 上一次 OCR 已经确认当前页没有可处理斯巴鲁:
    StopWithReset(斯巴鲁列表结束)

else if 当前可见范围已知出现状态 0:
    StopWithReset(已经到其它制造商或未知区)

else if 虚拟表满足完成边界:
    StopWithReset(没有剩余状态 3，最后一个 2 后面是 1)

else if 当前可见范围还有未知格子:
    Observe(当前页未完整观察)

else if 当前没有 3，且从当前 offset 开始有连续已知非 3 列:
    Scroll(连续已知列数量 - 最左侧保留的 1 列已知格子)

else:
    Scroll(1)
```

重要约束：

- 当前可见有 `3` 时，绝不能滚动。
- 已知后方有 `3` 时，滚动目标是“能看到这个 `3`”，不是固定滚动一格。
- OCR 只负责填表，不直接决定是否选车或滚动。
- 滚动动作的 tick 数在执行前由虚拟表算好。
- 停止动作只在没有进入固定点技能点序列时执行 `Esc` 复位。
- 如果固定点技能点序列完成后确认完成边界，则直接停止，不额外 `Esc`。

## 写表和已知格子

继续保留现有策略：

- 全局第一个格子是占位车，不点技能点，不删车。
- OCR 写表时忽略所有已经存在于虚拟表里的已知格子，避免反复写已经确认过的车辆状态。
- 激进滚动保留左侧 1 列已知格子。

但这些规则只能影响“哪些格子可信”和“滚动 tick 数”，不能在 OCR 或滚动 helper 内部偷偷改下一步流程。

用户修正原话：

```text
跟第一列没关系，总之就是OCR自动写表不会反复写已知格子
```

落地解释：

- “已知格子不反复写表”是 OCR 写表规则，不是 Planner 查询规则。
- 删除车辆导致的虚拟表位移由删除模块修改表格，不属于 OCR 写表模块。
- 如果某个格子在虚拟表内已经有状态，OCR 写表必须跳过这个格子。
- 如果某个格子还没有写入过，为了扩展虚拟表或恢复空洞，允许 OCR 写入一次；之后再次成为已知格子时就不再反复写。
- OCR 识别、OCR 坐标映射、OCR 写表策略要拆成独立模块被调用，不能和“下一步选车 / 滚动 / 停止”的决策混在一起。

补充约束：

- “最左侧保留 1 列已知格子”只影响滚动策略和“未知内容是否可信”，不能限制 Planner 选择已经写入虚拟表的状态 `3`。
- 如果状态 `3` 已经存在于虚拟表，并且当前可见，哪怕它位于最左侧保留的已知列，也必须执行 `Select`。
- 唯一不能选择的是全局第一个占位格 `(row=0, col=0)`。
- 同一格同时识别到 `全新` 和 `600` / `900` 时，自动点技术点阶段必须按状态 `3` 处理。状态 `4` 只能来自“没有有效 `全新` 的目标车”，不能覆盖 `3`。

## 落地步骤

1. 新增自动点技术点专用 Planner。
2. 给 `VirtualVehicleList` 补充 Planner 需要的纯查询方法：
   - 当前可见范围是否完整观察。
   - 后方是否有已知状态 `3`，以及应该滚到哪个 offset。
3. `MainLoopOnce()` 删除旧的 `FindFirstVehicle()` 预扫描入口。
4. 重写 `FindValidNewCell()`：
   - 每轮调用 Planner。
   - 根据 Planner 输出执行一个动作。
   - OCR 动作只写表，然后进入下一轮重新规划。
5. 保留现有 OCR、固定按键序列、扣技术点和完成后直接停止逻辑。

## 检查点

- 首屏 OCR 后如果出现状态 `3`，下一轮必须直接 `Select`，不能滚动。
- 如果已知状态 `3` 处在当前可见最左侧保留的已知列，下一轮仍然必须直接 `Select`，不能因为这列是 OCR 保留列就滚过去。
- 如果 OCR 摘要显示 `3>0`，写入虚拟表后也必须能查询到状态 `3`；不能被 `600` / `900` 写成 `4` / `5` 后滚走。
- 当前可见全是已知 `1/2`，后方虚拟表有 `3` 时，必须直接跳到能看到 `3` 的 offset。
- 当前可见没有 `3` 且有未知格子时，必须 OCR，不先滚动。
- 当前页开头出现状态 `0` 或 `1` 不能直接停止。必须继续 OCR/滚动，直到已经观察到目标车型区间，并确认目标车型区间末尾的下一个格子是 `0` 或 `1`，才能通过统一复位路径停止。
- 固定序列完成后确认完成边界时，不额外按 `Esc`。
