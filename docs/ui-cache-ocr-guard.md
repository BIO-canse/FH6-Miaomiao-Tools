# UI 坐标缓存 OCR 保险机制

## 用户原话

“OCR失败后主程序会退出，就是说那些依赖缓存点击的地方，同时拉独立子程序（防止阻塞，但是一般的OCR就正常阻塞也行）也截个图，如果能成功OCR代表没有错误，就无事发生，如果不能的话代表前面流程出错了，自动退出。”

补充要求：“其实这个较早失败就应该直接强制拦下停止的，不应该继续运行。”

后续纠正：“复用缓存的时候，为什么要等这么久？是不是你在等OCR返回？不要等啊，正常工作，等返回的时候再判断停不停”

## 目标

UI 坐标缓存用于加速“找文字并点击”的固定菜单入口，例如创意中心、最新最热、我的收藏等。缓存命中时会启动独立 OCR 校验进程，但主流程不能等待 OCR 识别完成。

新机制要求：缓存点击前启动独立校验子程序截图并 OCR 当前界面。主程序只短暂等待截图标记，尽量确保校验拿到的是点击前页面；随后正常点击并继续流程，不等待 OCR 返回。后续在阶段等待、子程序轮询、下一次缓存点击等检查点轮询校验结果；如果校验子程序找不到本次缓存对应的 UI 文字，主程序报错退出，并把截图、OCR 文本和错误写入 `debug/`。

## 设计

1. `FindTextAndClick` 只有在缓存命中时触发保险校验；正常 OCR 找文字点击仍沿用原阻塞流程。
2. 主程序在点击缓存坐标之前启动 `FH6UiCacheOcrGuard.exe`。
3. 校验子程序启动后立即绑定当前前台窗口，隐藏不了主程序叠加层，因此它只负责全屏/窗口客户区截图和 OCR，不发送任何输入。
4. 主程序只短暂等待“截图已完成”标记，尽量确认校验进程取到的是点击前页面；如果标记未及时返回，也继续主流程并后台等待结果。
5. 主程序不等待 OCR 识别完成，直接按原缓存坐标点击并继续流程。
6. 主程序在后续 `FullAutoSleep`、阶段间隔、子程序等待循环和下一次缓存点击前轮询校验子进程：
   - 退出码 `0`：校验通过，移除记录。
   - 非 `0`：读取结果文件，抛出异常，主程序按现有错误日志机制退出。
7. 子程序等待循环也会轮询校验子进程；如果历史遗留的校验失败结果在子程序运行期间返回，主程序会杀掉当前子程序并退出。
8. 急停会把校验子进程也纳入自动化进程集合一起杀掉。

## 输出

校验子程序会写：

- `state/ui-cache-guard-*.captured`：截图已完成标记。
- `state/ui-cache-guard-*.result.txt`：校验结果。
- `debug/ui-cache-guard/`：失败截图、OCR 响应和候选文本。

失败判定是“没有 OCR 到本次缓存对应文字”。例如复用“创意中心”的缓存坐标时，校验 OCR 必须能找到“创意中心”；否则说明当前页面不是预期页面，主程序退出。
