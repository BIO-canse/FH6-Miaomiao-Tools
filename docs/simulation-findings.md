## 推演程序

源码：`sim-cs/VirtualListSimulator.cs`

运行：

```bat
run_virtual_list_simulator.cmd
```

推演模型：

- 同时维护一份实际列表和一份 `VirtualVehicleList`。
- 假 OCR 只能读取当前可见范围。定表阶段第一次 OCR 会忽略可见第一列，因为这列是向前翻一列后露出的上一制造商列。
- 假 OCR 结果转换为 `ApplyFullObservation()` 的 `targets / validNew / deletable / drive / manufacturer / performanceScores` 集合；当前规则不再写入状态 `5`，`drive` 只用于叠加层标出当前可见的 `900` 分状态 `2` 候选。
- Planner 决策顺序按当前源码照搬，不改规则。
- OCR 写表继续遵守“已知格子不覆盖”。
- 技能点动作、删车动作直接修改虚拟表，因为它们不是 OCR。
- 实际列表增加车型段：普通斯巴鲁干扰车、目标车型前一位的锚定车型、目标车型连续段、跨制造商 `0`。
- 买车成功后直接按排序规则把状态 `3` 追加到虚拟表，同时推演里的实际列表也执行同样追加；后续轮次不重新建表。
- 阶段衔接要模拟真实流程：自动点技能点 -> 自动删车、自动删车 -> 找 `900` 分开蓝图候选，都会重新进入斯巴鲁车辆列表并把虚拟列表 offset 复位到 `0`。
- 当前实现增加了第一轮“定表阶段”：总控从车库标准位进入斯巴鲁车辆列表，先按 `Left` 让上一制造商列出现在可见第一列，把虚拟 offset 临时标为 `-1`；第一次 OCR 只忽略这一个可见前置列，所以可见第 2 列会写入虚拟表全局第 0 列。后续按 `visibleColumns - 1` 激进滚动继续建表。PaddleOCR/旧 OCR 后续不再固定跳过第一列；MediaOCR 因为车辆格按列识别且第 0 列是已知重叠列，每一轮都跳过本地第 0 列写表。已知格仍然不反复写。直到目标车型段后出现非 `2/3/4/5` 边界，然后按 `Esc` 回到车库标准位。
- 定表阶段会记录每个格子的性能分；`3` 必须同时满足目标车型、全新、性能分 `600`。
- 买车成功后不再等待 OCR 发现新车，而是直接把买到的车作为性能分 `600` 的状态 `3` 插入虚拟表：有 `3/4` 就跟到最后一个 `3/4` 后；没有 `3/4` 就按性能分插入。
- 状态码以用户最新原话为准：`-1=空格`，`0=其他厂商`，`1=斯巴鲁的车`，`2=指定型号`，`3=指定型号&600&全新`，`4=指定型号&600&非全新`，`5=旧版开蓝图标记，当前不再写入`。`900` 现在是正常 `2` 车，开蓝图车辆通过 `PerformanceScore == 900` 且状态 `2` 选择列表排序最前的指定车型。
- 写表优先级：OCR 可以直接写状态 `4`，因为车库里可能本来就有旧车；但 `目标车+600+全新` 必须写成状态 `3` 并且不能进入删车目标，只有 `目标车+600+没有全新` 才写成状态 `4`。
- 定表完成后，自动点技术点、自动删车、找 `900` 分开蓝图候选都进入表驱动模式：只允许选择、滚动、停止，不允许再触发车辆格 OCR。若 Planner 走到 Observe 分支，视为定表不完整并报错。
- 虚拟表现在允许跨轮使用；后续轮次不重新定表，依靠点技能、删车、买车动作直接修改并持久化同一张表。
- 表驱动退出条件简化：自动点技术点只要表内没有 `3` 就退出；自动删车只要表内没有 `4` 就退出；找开蓝图车不做自动退出，因为会从表内选择列表排序最前的 `900` 分状态 `2` 指定车型。

## 当前覆盖场景

截至 2026-05-24，`run_virtual_list_simulator.cmd` 覆盖以下场景，当前结果 `findings=0`：

1. 两轮完整流程：干扰车 + 目标连续段 + 买车追加到锚定车型后。
2. 技术点中途不够：第一轮只点一辆，刷到 `999` 后第二轮继续。
3. 开局当前可见页面没有 `2/3/4`，目标在后方。
4. 没有目标车型：普通斯巴鲁后滚到跨制造商 `0` 后停止。
5. 开局技术点为 `0`：有 `3` 也不能点，不能误删或误滚成完成。
6. 删车位移：连续 `4` 被删除后列表自动前移，后方 `4` 不能漏。
7. MediaOCR 建表保留列：忽略第 `0` 列时不能把未识别列写成空格，第 `1` 列及后续仍要正常写入。

## 推演结论

### 阶段切换必须复位 offset

如果技能点阶段滚到后方，再把同一张虚拟表直接交给删车阶段，但不把 `CurrentOffset` 复位到 `0`，删车 Planner 可能会看不到位于当前 offset 之前的状态 `4`，随后一直向后滚。

当前实际代码中，衔接删车会重新进入斯巴鲁列表并调用 `EnterVehicleListAtCachedOffset()`，找 `900` 分开蓝图车辆前也会 `ReopenSubaruListFromVehicleListForDriveSearch()`，两处都会 `vehicleList.ResetView()`。因此模拟器也必须保留这个阶段切换复位，不然模拟结果会变成不符合真实流程的假失败。

## 已复现问题

### 1. 技能点后进入删车，虚拟表没有产生状态 4

场景：

```text
实际列表：
普通斯巴鲁车
斯巴鲁非目标车
多辆状态 3：目标车 + 全新
后面是状态 0 边界
```

推演结果：

```text
skill step=0 offset=0 action=Observe 当前可见目标段还有未知格子
skill step=1 offset=0 action=Select 当前可见范围已有状态 3
skill step=2 offset=0 action=Select 当前可见范围已有状态 3
skill step=3 offset=0 action=Select 当前可见范围已有状态 3
skill step=4 offset=0 action=Select 当前可见范围已有状态 3
skill step=5 offset=0 action=Select 当前可见范围已有状态 3
skill step=6 offset=0 action=Select 当前可见范围已有状态 3
skill step=7 offset=0 action=Stop 没有状态 3 且完成边界成立
delete step=0 offset=0 action=Stop 没有状态 4 且删车完成边界成立
```

历史问题：

技能点阶段点完一辆状态 `3` 后，实际车辆会进入后续删车要处理的状态；旧实现里 `MarkProcessed()` 只把该格从 `3` 改为普通目标车状态 `2`。

删车阶段读取这张本轮表后，因为状态 `2` 是已知格子，OCR 不会覆盖它，所以删车 Planner 看不到状态 `4`，直接按完成边界停止。

结论：

这不是 OCR 规则错，而是动作模块没有把“点完技术点后的格子状态变化”直接写进虚拟表。根据用户原话，删车和选车不属于 OCR，可以改已有表格；这个状态转移应由动作模块维护，不能指望后续 OCR 修正已知格子。

处理：

`MarkProcessed()` 应在固定点技术点序列成功后，把该格直接写成状态 `4`，而不是状态 `2`。建表/OCR 阶段也允许把已存在旧车识别为状态 `4`，但必须保证状态 `3` 的优先级高于状态 `4`。
