# 车辆列表辅助定位方案

当前全自动主链路第一轮先用 OCR 建立完整虚拟车辆表。定表完成后，自动点技术点、自动删车、自动买车追加和找 `900` 分开蓝图车辆都直接读写这张表，不再每轮车辆格 OCR。

## 用途

车辆列表里目标是定位 `IMPREZA 22B-STI`，然后进入该车的操作菜单和技术点页面。

这个方案只负责在车辆列表中定位目标车辆。

## 两层模型

1. `GridGeometry`：车辆列表的固定像素几何。
2. `VirtualVehicleList`：按“列内从上到下，满列后再开新列”的顺序保存车辆状态、当前 offset 和运行期编辑记录。

`GridGeometry` 在同一台电脑、同一分辨率、同一缩放设置下固定。车辆列表内容变化由定表、点技能点、删车和买车追加直接修改 `VirtualVehicleList`。

## GridGeometry

几何不再通过 OCR 车名推断。

首次没有 `config/user-settings.json` 时，程序会询问：

```text
visible_rows
visible_columns
```

然后弹出框选层，让玩家鼠标框选“所有完整可见车辆格子的整体区域”。程序按玩家输入的行列数切分单格。保存字段：

```text
grid_cell_left
grid_cell_top
grid_cell_width
grid_cell_height
```

后续直接构建格子：

```text
cellLeft = grid_cell_left + col * grid_cell_width
cellTop  = grid_cell_top  + row * grid_cell_height
hitRect[row][col] = cellLeft/cellTop/grid_cell_width/grid_cell_height
```

保存后几何立即锁定，不再滚动暴露三列车辆，不再根据 `IMPREZA 22B-STI` 字段估算格子大小。

如果框选时能够识别到框选区域下方的游戏窗口，设置文件会同时保存窗口客户区尺寸。后续运行时如果绑定窗口的客户区发生等比缩放，格子坐标会按客户区比例自动迁移；如果横向和纵向比例不一致，程序停止并提示重设设置重新框选。

## VirtualVehicleList

定表阶段负责把 OCR 结果写入虚拟列表：

1. 从车库标准位进入斯巴鲁车辆列表。
2. 第一次 OCR 前先按 `Left` 向前翻一列；这一次可见第一列属于上一制造商，不写入虚拟表。
3. 后续可见格子用车辆名、`全新`、`斯巴鲁` 和三位数性能分等 OCR 字段推导状态码与 `PerformanceScore`。
4. 一直激进滚动并 OCR，直到确认目标车型段后出现边界。

定表后，车辆内容变化由动作本身维护：点完技能点后状态 `3 -> 4`，删车后虚拟表前移，买车成功后按排序规则追加状态 `3`。车辆格 OCR 不再参与这些阶段。

## 运行期虚拟列表保存

`VirtualVehicleList` 会随着车辆列表横向翻列持续扩展，并实时覆盖保存到 `state/virtual-vehicle-list.json`。车辆列表内统一只用键盘移动：向后翻列用 `Right`，向前翻列用 `Left`，不再用鼠标滚轮操作车辆格子列表。

独立启动时不会读取这个文件，避免使用上次运行后的过期车辆顺序。全自动流程会在第一轮先执行定表阶段，之后点技术点、删车、买车和找 `900` 分开蓝图车辆都直接读写同一张虚拟表；买车成功后会按排序规则把新买的指定车型直接追加为状态 `3`，后续轮次不重新 OCR 扫车辆格。

虚拟列表规则：

- 定表阶段第一次 OCR 前会先按 `Left` 向前翻一列；这一次可见第一列属于上一制造商，不写入虚拟表。此时可见第 2 列会映射为虚拟表全局第 0 列。
- 全局第一个格子不再固定当作占位车；它如果是状态 `2` / `3` / `4`，也必须和其它格子一样参与处理。
- OCR 写入虚拟列表时忽略所有已经存在于虚拟表里的已知格子，避免反复写已经确认过的车辆状态。
- 用户修正原话：“跟第一列没关系，总之就是OCR自动写表不会反复写已知格子。”
- 全自动定表完成后，自动点技术点、自动删车和全自动找 `900` 分开蓝图车辆都只读写虚拟列表；如果这些阶段走到车辆格 OCR 分支，说明定表或状态维护不完整，应写日志并停止排查。
- 只有格子内识别到 `斯巴鲁` 的非目标车才记为状态 `1`。没有 `斯巴鲁` 的格子记为状态 `0`，代表其它制造商或未确认制造商，不再参与状态 `1/2/3/4` 的连续区间跳过。
- 如果当前列表区域没有识别到新的可处理 `斯巴鲁` 格子，脚本认为可能已经滚出斯巴鲁制造商区域或没有剩余车辆，最终仍以虚拟列表边界判断为准。
- 列表边界不能用“当前页第一个 `0` 或 `1`”判断。开头是 `0` 或 `1` 不能直接退；只有已经观察到目标车型区间后，目标车型区间末尾的下一个格子确认是 `0` 或 `1`，才算完成边界。自动点技术点必须同时满足“边界前没有 `3`”和“目标车型区间末尾后面是 `0` 或 `1`”才退出；自动删车必须同时满足“边界前没有 `4`”和“目标车型区间末尾后面是 `0` 或 `1`”才退出。
- 如果从列表开头一直没有找到目标车型状态 `2/3/4`，但连续观察到的列表末尾已经出现状态 `0`，也可以停止；这里不能把状态 `1` 当作停止边界，因为 `1` 仍然表示斯巴鲁车辆。
- 独立启动或缺少可信状态时，首轮先 OCR 当前屏幕；衔接启动时优先使用上一段脚本保存的虚拟列表状态，已有状态 `2` / `3` / `4` 可直接跳转或处理。
- 对开蓝图车辆选择来说，三位数性能分只写入 `PerformanceScore`。`900` 也是普通状态 `2` 指定车型，不能覆盖 `2/3/4`；选择目标时只选择状态 `2` 且 `PerformanceScore == 900` 的格子，多个候选按列表顺序选择最前的一个。
- 激进滚动的目标是显示尽可能多的未知列，同时最左侧保留 1 列已知格子。
- 如果连续已知区长度为 `run`，激进滚动滚动 `run - 1` 列；滚动后未知列从第 2 列开始进入 OCR 范围。

状态码：

```text
-1 = 空格，格子区域内没有识别到车辆内容
0 = 其它制造商
1 = 斯巴鲁的车，但不是指定型号
2 = 指定型号
3 = 指定型号 & 600 & 全新
4 = 指定型号 & 600 & 非全新，可删车辆
5 = 旧版开蓝图车标记，当前不再写入虚拟表
```

用户修正规则：“5的规则改一下，现在不要直接写5（避免覆盖掉），而是选择性能分（一定是三位数）最高的指定类型车辆”。后续修正：“不要选择最高性能分得了，只选择列表排序最前的性能分为900的2。”当前实现只把三位数性能分保存到 `PerformanceScore`，不再把 `900` 车辆写成状态 `5`；开蓝图车由 Planner 选择列表排序最前的 `900` 分状态 `2` 指定车型。

## 选择目标车辆

如果屏幕上有多个有效目标车辆格子，选择最左最上的一个：

```text
rank = col * 1000 + row
```

方向键移动现在按游戏实际行为处理：左右键会触发车辆列表横向滚动，选中框仍停在屏幕第一列；上下键只改变当前选中行。

因此选择目标时，Planner 只决定要处理哪个全局格子；执行器一次性生成键盘路径：

```text
horizontalDelta = targetGlobalCol - CurrentOffset
Right/Left x abs(horizontalDelta)
Down  x targetRow
Enter
```

左右移动后，目标全局列会成为当前第一列；后续点技术点、删车等表格写入都以新的第一列本地格子为准。

`targetRow` 必须在 `0` 到 `visible_rows - 1` 之间。

## 叠加层

如果启用车辆列表辅助定位，叠加层显示：

- 蓝色边框：车辆列表格子。
- 无填充：空格子、状态 `1`、或 `全新` 但不属于目标车。
- 一条蓝色斜线：状态 `2`，指定型号。
- 两条蓝色斜线：状态 `3`，指定型号且有有效 `全新`。
- 两条蓝色斜线加横竖等分线：状态 `4`，可删车辆。
- 蓝色小圆：当前选中的开蓝图车辆，也就是列表排序最前的 `900` 分状态 `2` 指定车型；这是叠加层高亮，不代表虚拟表状态被写成 `5`。
- 青绿色细框：OCR 识别到的车名字段。
- 橙色细框：OCR 识别到的 `全新` 字段。
- 紫色细框：OCR 识别到的 `斯巴鲁` 字段。
- 每个格子显示 `G全局列,R行 S状态码`，滚动后 `G` 会跟随 `CurrentOffset` 更新，未知格子显示 `S?`。
- 左上角状态：运行模式、大阶段、当前操作、下一步/动作串、轮次、技术点、超级抽奖、刷点进度、虚拟表摘要、总耗时、失败次数。

截图前必须隐藏叠加层，避免污染 OCR。
