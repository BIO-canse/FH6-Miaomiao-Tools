# 当前完整流程规格

## 默认规则

- 打开程序后先选择运行模式：`1` 正常模式，`2` 调试模式，`3` 重设设置。
- 如果选择 `3`，删除 `config/user-settings.json`，重新询问玩家“我的车辆页面中能够看到完整的几行、几列车辆”，重新框选完整可见车辆格子的整体区域，然后回到运行模式菜单。框选页面参考 `assets/vehicle-grid-page.png`。
- 如果选择 `1` 或 `2`，再检查 `config/user-settings.json`。
- 如果该文件不存在，询问玩家“我的车辆页面中能够看到完整的几行、几列车辆”。只统计右侧完整车辆格子；左侧车辆详情栏、最左/最右边缘露出的半个格子都不要算。
- 如果该文件来自旧版本，缺少 `dpi_aware_coordinates` 或 `calibration_mode=full_grid_v1` 标记，也会作废并重新框选，避免 Windows DPI 缩放或单格框选误差导致截图区域偏移。
- 支持任意分辨率，前提是首次运行或重设设置时按当前画面正确框选完整车辆格子区域。玩家更改分辨率、窗口缩放或 UI 比例后，需要重新选择 `3` 重设设置。
- 启动确认后脚本会绑定目标窗口客户区，截图、OCR、叠加层和鼠标避让坐标都以这个窗口为基准。框选设置会保存当时的窗口客户区；如果后续窗口分辨率等比变化，虚拟表格映射会自动等比迁移；如果窗口比例非等比变化，程序提示重设设置重新框选。
- 建议玩家把画质调到最低或较低，减少菜单动画和加载耗时，避免性能差的电脑超过脚本预留等待时间。
- 玩家需要输入当前 CR 点数并预留足够余额。全自动流程买车每辆固定按 `86000 CR` 扣除；每次买车前都会检查余额，CR 不足时不会继续按购买键，会返回菜单并自动退出。
- 刷蓝图车辆必须是 `900` 分的 `IMPREZA 22B-STI VERSION`。推荐调教代码 `122 378 124`；也可以使用其它能稳定跑到指定时间范围的调教，但不要盲目追求最快，太快会因为倍率到不了 `9` 导致少拿技术点，太慢也会错过脚本时序。程序只选择列表排序最前的 `900` 分状态 `2` 指定车型进入蓝图。
- 蓝图代码 `890 169 683` 必须已经加入“我的收藏”，并且处于“我的收藏”的第一位。程序进入收藏蓝图列表后会直接选择第一项，参考 `assets/favorite-blueprint-first.png`。
- 刷一次蓝图后，游戏结算会显示实际局内用时。实际局内用时必须大于 `24` 秒且小于 `28` 秒；太快会导致倍率不到 `9`，太慢会错过脚本时序。脚本会从按下 `Enter` 开始计时，等待 `37` 秒后按 `X`，这个实际等待保持不变。
- 车库里还需要至少有一辆“所属制造商排序在斯巴鲁后方”的任意车辆。脚本需要靠它确认列表已经滚出斯巴鲁制造商区域；如果斯巴鲁后面没有任何其它制造商车辆，程序可能无法判断斯巴鲁区域已经结束。
- 建议先手动买几辆不同的斯巴鲁车辆，并且多买一些用于刷技术点的指定车型。滚出制造商边界这件事不读取游戏内存，只能依赖 OCR 和虚拟列表判断；斯巴鲁车辆太少时更容易碰到边界判断问题。
- 确认开始后的 `10` 秒等待结束时，游戏必须处于“大世界”可操作状态，参考 `assets/overworld-start.png`。
- 车库里有其它车辆或干扰车辆不会影响流程；脚本依赖首次框选和 OCR 动态识别，只要求前提条件满足。
- 然后弹出框选层，要求玩家用鼠标框选“所有完整可见车辆格子的整体区域”：从最左上完整格子的左上角拖到最右下完整格子的右下角。不要框左侧车辆详情栏、顶部制造商标签栏、底部按键提示，也不要框边缘不完整的半格。
- 弹出框选层前需要按 `Enter` 确认；确认后控制台隐藏，再开始框选。
- 框选层会同步显示起点、当前拖动点和蓝色矩形。
- 保存 `visible_rows`、`visible_columns`，并用整体区域按行列数切分出单格 `left/top/width/height` 后继续。
- 如果该文件存在，后续直接使用保存的行列和格子尺寸。
- 每次总控启动会询问玩家当前技术点总数；该值不是永久配置，运行中会写入 `state/` 里的计数文件，作为总控和子程序同步用的临时状态；直接回车默认 `999`。
- 每处理完一辆车，技术点总数扣 `32`。
- 全自动刷技术点子程序运行期间，总控浮层显示当前技术点数量、剩余轮数、预计剩余时间和预计刷满时间；`MinuteWLoop.exe` 每完成 1 轮写回一次计数，总控随之刷新显示。预计每轮按 `48.3` 秒计算：`Enter 0.1 秒 + 37 秒 + X 0.1 秒 + 1 秒 + Enter 0.1 秒 + 10 秒`。
- 技术点计数必须有事件日志。每次自动点技能点成功后记录 `-32`，超级抽奖计数 `+1`；每次 `MinuteWLoop.exe` 完成 1 轮后记录 `+10`。日志要保留 before/after、delta、reason、轮次、时间戳，方便还原“先一串 -32、再一串 +10”的真实顺序。
- 全自动浮层显示以可操作信息为主：当前大阶段、当前操作、下一步/下一串动作、主循环轮次、技术点数量、已获得超级抽奖数量、刷点剩余轮数和预计刷满时间。几何、校准、OCR 等细节不再常驻显示，除非后续进入专门调试视图。
- 总耗时按程序启动时的系统时间和当前系统时间计算，而不是依赖用户不可见的内部估算。
- 每轮主循环开头先检查剩余技术点；如果剩余技术点小于 `32`，脚本自动停止。
- 普通模式启动后等待 `10` 秒，然后进入主循环。
- 调试模式跳过开局 `10` 秒等待，直接进入主循环。
- 调试模式会保存每次 OCR 的截图链路到 `debug/screenshots/`，包含原始截图、实际送入 OCR 的缩放图、OCR 原始返回和元信息。
- 下一次调试模式启动时，会先清空 `debug/screenshots/`。
- OCR 后端默认使用 PaddleOCR PP-OCRv5；普通模式截图只在内存中传给 OCR，不保留调试截图。
- 用户原话：“急停检测做成完全独立的进程，这玩意作用就是拿到急停快捷键后后直接停掉所有进程。这样我们就不用担心阻塞了，OCR如果为了稳定性阻塞一下就阻塞吧”
- 立即急停由独立监控进程负责检测 `Space+C`。监控进程与业务脚本分离，检测到后直接结束当前工具包内的所有自动化进程和打包 Python/OCR 子进程，所以 OCR 调用为了稳定性可以正常等待返回，不再依赖业务线程在 OCR 等待中轮询。
- 未特别说明时，鼠标点击、键盘按键按下时间都是 `0.1` 秒。
- 未特别说明时，连续相同操作间隔都是 `0.1` 秒。
- 立即急停快捷键：`Space+C`，直接退出当前脚本。
- 安全急停快捷键：`Space+V`。如果当前在主程序且没有子程序运行，直接退出；如果子程序正在运行，主程序只向子程序发送安全退出请求，等子程序完成当前轮并复位后退出主程序。

## 主循环

### 1. 进入制造商列表并选择斯巴鲁

保留主循环最前面的 `Enter`。

1. 按 `Enter`。
2. 等待 `1` 秒。
3. 按 `Backspace`，打开制造商页面。
4. 等待 `0.5` 秒，让制造商页面稳定。
5. 鼠标移动到屏幕中心，保证滚轮作用在制造商列表上。
6. 鼠标滚轮向下滚动 `10` 格。
7. 整屏 OCR 找 `斯巴鲁` 并点击，进入斯巴鲁制造商车辆列表。

这一段已经替换掉原来的方向键录制/回放和更早的图寻制造商逻辑：

```text
Enter
寻找“图一标识”
鼠标移动到对应 UI 按键
循环点击并寻找“斯巴鲁”
找到后再次点击进入斯巴鲁车辆列表
OCR 找斯巴鲁
```

除此以外，后续流程保持原定逻辑。

### 2. 找到第一处 IMPREZA 22B-STI

1. 开始寻找 `IMPREZA 22B-STI`。
2. 每轮先 OCR 识别 `IMPREZA 22B-STI`。
3. 如果没找到，按 `Right` 向后翻一格。
4. 找到后停止滚动。
5. 如果找到多个，按“优先左，其次上”选择：

```text
rank = x * 1000 + y
```

注意：找到的这一轮不再滚动。

## 虚拟格子算法

`GridGeometry` 不再通过 OCR 车名推断格子大小，也不再为了暴露三列指定车辆而额外滚动。

几何信息来自首次设置时玩家框选的最左上角完整格子：

```text
cellLeft = grid_cell_left + col * grid_cell_width
cellTop  = grid_cell_top  + row * grid_cell_height
hitRect[row][col] = cellLeft/cellTop/grid_cell_width/grid_cell_height
```

`visible_rows` 和 `visible_columns` 也来自首次设置。99% 情况下行数是 `3`，但允许玩家输入其它行数以兼容特殊分辨率比例。

保存后几何立即锁定。后续只更新格子内容，不重新建表、不重新估算格子尺寸。

## 后续循环识别

建好 `GridGeometry` 后，后续不再重新建表。

运行期虚拟列表状态：

```text
-1 = 空格
0 = 其他厂商
1 = 斯巴鲁的车
2 = 指定型号 IMPREZA 22B-STI
3 = 指定型号 & 600 & 全新，代表待点技术点
4 = 指定型号 & 600 & 非全新，代表点完技术点后等待删车
5 = 旧版开蓝图车标记，当前不再写入虚拟表
```

独立启动时不会读取旧虚拟列表文件；全自动流程第一轮会先执行定表阶段并覆盖 `state/virtual-vehicle-list.json`。之后自动点技能点、自动删车、买车追加、找 `900` 分开蓝图车辆都直接读写这张表；后续轮次不重新 OCR 扫车辆格。

用户修正规则：“5的规则改一下，现在不要直接写5（避免覆盖掉），而是选择性能分（一定是三位数）最高的指定类型车辆”。后续修正：“不要选择最高性能分得了，只选择列表排序最前的性能分为900的2。”落实规则：定表 OCR 只记录指定车型和三位数性能分，不再把 `900` 写成状态 `5`；开蓝图时只选择状态 `2` 且 `PerformanceScore == 900` 的格子，如果有多个则按列表顺序选择最前的一个。

如果虚拟列表里已经有 `3`，下一轮进入斯巴鲁列表后优先直接用键盘 `Right` 翻到能看到这个 `3` 的位置，然后按方向键选中处理，不做 OCR。

固定操作完成后，本次处理的格子从 `3` 变成 `4`，进入后续自动删车阶段。

如果固定按键序列完成后已经能确认完成边界，例如最后已知列是 `331`，脚本会直接停止，不额外按 `Esc`，因为固定序列本身已经复位。

如果进入下一轮后虚拟表内已经没有状态 `3`，这时脚本没有执行点技能点固定序列，需要额外按 `Esc`，等待 `1` 秒回到标准位，然后自动停止。

自动点技术点停止规则：

1. 全自动流程使用定表虚拟列表。
2. 虚拟列表里存在状态 `3` 时，必须选择最左最上的状态 `3` 并处理。
3. 状态 `3` 可能在当前可见范围，也可能在后方；执行器根据全局列和当前 offset 生成键盘路径。
4. 虚拟列表里不存在状态 `3` 时，自动点技术点停止。
5. 如果刚点完最后一辆车后停止，不额外按 `Esc`，因为固定点技能点序列已经复位。

状态 `0` 是边界信号，但在全自动表驱动阶段只由定表阶段负责确认。定表完成后，点技术点、删车、买车追加、找 `900` 分开蓝图车辆都通过虚拟表状态变化维护车辆顺序，不再每轮车辆格 OCR。

选择有效目标时，如果有多个状态 `3`，按格子坐标选最左最上的：

```text
rank = col * 1000 + row
```

## 方向键移动到目标格

默认选中位置是当前屏幕从左到右第一列、第一行。

虚拟表格是 `visible_rows` 行、`visible_columns` 列：

```text
targetCol = 目标列，从 0 开始
targetRow = 目标行，从 0 开始，最大值是 visible_rows - 1
```

车辆列表内统一只用键盘移动。方向键移动按当前游戏行为处理：左右方向键会触发车辆列表横向滚动，选中框仍停在屏幕第一列；上下方向键只改变选中行。这里的“滚动/翻列”都指按 `Right` / `Left`，不再用鼠标滚轮操作车辆列表。

选择目标时，Planner 只决定“要处理哪个全局车辆格子”，不负责移动。执行器根据当前 `CurrentOffset` 和目标全局列一次性生成键盘路径：

```text
horizontalDelta = targetGlobalCol - CurrentOffset
如果 horizontalDelta > 0：Right x horizontalDelta
如果 horizontalDelta < 0：Left  x abs(horizontalDelta)
Down  x targetRow
Enter
等待 1 秒
```

左右方向键执行后，虚拟列表同步更新 `CurrentOffset`，此时目标会落在屏幕第一列。点技术点、删车、找 `900` 分开蓝图车辆都按这个“全局目标 -> 一次性路径”的模型执行，避免出现“先滚到目标可见，又在选中阶段多 Right 一格”的叠加错误。

每次方向键按下 `0.1` 秒，连续方向键间隔 `0.1` 秒。`Enter` / `Esc` 这类会打开或关闭菜单的操作后面至少等待 `1` 秒。

目标必须在当前屏幕可见范围内；如果不在，继续滚动寻找，不执行方向键移动。

## 当前关键假设

- 主循环开头的 `Enter` 保留。
- 制造商定位段使用 `Enter -> 等待 1 秒 -> Backspace -> 等待 0.5 秒让制造商页面稳定 -> 鼠标移到屏幕中心 -> 向下滚动 10 格 -> OCR 点击斯巴鲁 -> 等待进入斯巴鲁列表 -> 鼠标停到第二行最后一列右侧屏幕边缘`。
- 虚拟格子几何来自首次鼠标框选的完整可见车辆格子整体区域，不再用 OCR 推断。
- 定表阶段第一次 OCR 前会先按 `Left` 向前翻一列；这一次可见第一列属于上一制造商，不写入虚拟表。此时可见第 2 列会映射为虚拟表全局第 0 列。
- 全局第一个格子不再是永久占位车；如果它是状态 `3` / `4` / `5`，也必须正常处理。
- 定表阶段 OCR 自动写表不会反复写已知格子；删除车辆导致的表格变化由删除模块修改表格，不属于 OCR 模块。
- 用户修正原话：“跟第一列没关系，总之就是OCR自动写表不会反复写已知格子。”
- “最左侧保留 1 列已知格子”只影响滚动策略和未知内容扩展，不影响已经写入虚拟列表的状态 `3`。如果已知状态 `3` 当前可见，即使位于这列，也必须直接选中处理。
- 同一车辆格子同时识别到 `全新` 和 `600` / 其它三位数性能分时，自动点技术点阶段按状态 `3` 处理；状态 `4` 只能来自没有有效 `全新` 的目标车。性能分只作为开蓝图车选择依据，不能覆盖状态 `2/3/4`。
- 激进滚动的目标是让当前屏幕显示尽可能多的未知列，同时最左侧保留 1 列已知格子。
- 如果从当前 offset 开始的连续已知区长度为 `run`，实际滚动 `run - 1` 列；滚动后最左列是最后一列已知列，未知列从第 2 列开始进入 OCR 范围。
- OCR 发生时，叠加层会用不同颜色细框标出本轮识别到的符合条件字段，便于检查车名、`全新`、`斯巴鲁`、`600` 和三位数性能分是否识别正确。
- 格子绝对像素坐标在同一电脑上固定。
- 全自动定表完成后，车辆内容变化由点技能点、删车、买车动作直接修改虚拟表，不再靠每轮车辆格 OCR 更新屏幕占用表。
- OCR 失败时会把本轮识别文本和坐标写入 `debug/ocr-last-*.txt`，用于调整识别规则。
